# 富文本顶层标题拆分实施记录

## 当前状态

Tiptap 富文本笔记现已支持按根节点 H1/H2 拆分，并始终保持 `tiptap-json` 格式，不经过 Markdown 中转。

已实现范围：

- 按顶层 H1/H2 生成拆分预览；
- 选择全部或部分章节；
- 在服务端基于最新正文和 `version` 重新规划；
- SQLite 事务创建章节笔记并更新原笔记；
- 标签、归档状态和目标笔记本权限继承；
- 附件元数据拆分，物理文件继续去重；
- Tiptap 目录 `link` mark 与 `note_links` 索引；
- 即时撤销并恢复原始 Tiptap JSON；
- 旧 `noteId + blockId` 链接按拆分记录动态重定向；
- 非法、历史或不完整 JSON 的安全拒绝；
- 独立前后端回归测试和 Note Split CI。

## 富文本规划规则

规划器只读取 Tiptap 文档根节点的 `doc.content`：

1. 仅将根节点中 `type === "heading"` 且 `attrs.level` 精确等于用户选择层级的节点视为边界。
2. 列表、引用、表格、折叠块等嵌套内容中的标题不作为拆分边界。
3. 标题文本递归收集其 `text` / `hardBreak` 子节点，仅用于章节笔记标题。
4. 章节正文为标题节点之后、下一个同级标题之前的完整节点数组。
5. 节点、`attrs`、`marks` 和未知扩展字段原样保留。
6. 标题节点本身不复制到章节正文，与 Markdown 拆分语义一致。
7. 只有标题而没有正文的章节会生成一个空 `paragraph`，避免产生无法挂载的空文档。

子笔记保持标准 Tiptap 文档结构：

```json
{
  "type": "doc",
  "content": [
    {
      "type": "paragraph",
      "content": []
    }
  ]
}
```

## 原笔记目录结构

原笔记使用标准 Tiptap 节点构造目录：

1. 用户选择保留的前言节点；
2. 拆分说明 `blockquote`；
3. 比拆分层级低一级的目录 `heading`；
4. `orderedList` 目录；
5. 未选择的原始章节节点。

目录链接使用标准 `link` mark：

```json
{
  "type": "text",
  "text": "章节标题",
  "marks": [
    {
      "type": "link",
      "attrs": {
        "href": "note:<uuid>",
        "target": null,
        "rel": "noopener noreferrer nofollow nowen-title-alias"
      }
    }
  ]
}
```

目录标题必须低于拆分层级。例如按 H2 拆分时生成 H3，避免下次扫描把“目录”识别为用户的同级章节。

操作 ID 仅保存在 `note_split_operations`，不会作为普通段落写进富文本正文。

## 事务与数据安全

服务端不会接收浏览器提交的章节节点内容，只接收：

- `version`；
- `headingLevel`；
- `sectionIndexes`；
- 目标笔记本；
- 是否保留前言；
- Block 引用风险确认标记。

服务端基于数据库中的最新 Tiptap JSON 重新规划，并在一个 SQLite 事务中完成：

- 写入拆分前版本历史；
- 创建章节笔记；
- 继承标签和归档状态；
- 拆分附件元数据；
- 更新全文、Block、附件和双链索引；
- 更新原笔记目录或保留章节；
- 写入可撤销操作记录。

原笔记更新使用版本条件：

```sql
UPDATE notes
SET ...
WHERE id = ? AND version = ?
```

未命中时整个事务回滚，不留下半完成章节。

## 附件策略

附件不会重复写入磁盘：

- 原笔记仍引用附件时，原附件 ID 保留在原笔记；
- 新章节创建新的附件元数据 ID，并复用同一个物理 `path`；
- 原笔记不再引用附件时，首个章节可以接管原附件 ID；
- 其他章节继续创建同路径元数据副本；
- 撤销时迁回原附件并删除拆分产生的元数据副本。

因此删除目录页或章节不会让其他仍在使用同一物理文件的笔记产生裂图。

## 撤销边界

富文本拆分可以即时撤销，但必须同时满足：

- 原笔记没有继续编辑；
- 所有章节笔记仍存在；
- 章节笔记没有继续编辑或进入回收站；
- 章节没有上传新附件；
- 拆分操作尚未撤销。

撤销会恢复操作记录中的原始 `tiptap-json`，而不是重新拼接章节，避免节点顺序、属性或未知扩展字段丢失。

## Block 链接重定向

拆分前的外部链接可能仍保存为：

```text
note:原笔记ID#blk:<blockId>
```

系统不会批量改写所有引用笔记的正文。打开带 Block 的内部链接时，前端先调用 `/api/blocks/resolve`，后端按以下顺序解析：

1. 原笔记中仍存在该 Block：继续打开原笔记并定位；
2. Block 已被拆分：读取 `note_split_operations` 的原始快照；
3. 根据原章节的稳定 Block ID 与 `note_split_items` 重建章节映射；
4. 正文 Block 跳转到新章节中的同一 Block；
5. 被省略的章节标题 Block 跳转到章节笔记顶部；
6. 章节再次拆分时，最多继续解析 8 跳；
7. 拆分撤销后，`status = undone` 的记录不再参与解析，原 Block 恢复后自动优先命中。

重定向过程不会返回内部操作 ID或完整拆分链。源笔记、中间章节和最终章节都必须通过读取权限校验，避免链接解析绕过 ACL。

### 明确限制

以下极端情况不会猜测目标：

- 多个被选章节标题完全相同；
- 这些章节都没有正文 Block 可用于稳定匹配；
- 仅凭旧操作记录无法唯一确定目标章节。

此时返回 `BLOCK_NOT_FOUND`，优先保证“不跳错”。普通含正文的章节、标题唯一的空章节以及二次拆分链均可解析。

拆分前的风险检测和二次确认仍然保留，用于让用户知道外部 Block 引用会依赖重定向，而不是正文被永久改写。

## Y.js 边界

当前后端 Y.js 服务维护的是 Markdown `Y.Text`。富文本拆分不得把 Tiptap JSON 写入该房间，否则空闲回写可能把富文本内容按 Markdown 覆盖。

因此富文本拆分链路：

- 依赖前端强制保存后重新读取 REST 权威版本；
- 使用 SQLite 事务更新 Tiptap JSON；
- 发送笔记列表和版本变更通知；
- 不调用 Markdown 的 `yFlush`、`yReplaceContentAsUpdate` 或 Y.Text 广播。

后续若富文本协同迁移为独立的 ProseMirror/Y.js 结构，需要为富文本拆分提供格式感知的 CRDT 替换协议。

## 已覆盖测试

- 仅顶层 H1/H2 生效，嵌套标题不切分；
- marks、引用、未知节点和根文档属性原样保留；
- 选择部分章节时未选节点继续留在原笔记；
- 标题为空或章节正文为空时仍生成合法文档；
- 同一附件被已选和未选章节引用时不失效、不重复写盘；
- 目录 `link` mark 可生成真实 `note_links`；
- 外部 Block 引用必须二次确认；
- 第一次风险响应不产生任何拆分记录；
- 事务拆分和即时撤销；
- 撤销后恢复原始 JSON、附件和章节状态；
- Tiptap 与 Markdown Block 重定向；
- 标题 Block 跳转章节顶部；
- 连续两次拆分的多跳解析；
- 客户端解析失败时回退旧行为；
- 非法 JSON 和不满足标题数量的文档被拒绝。

## 后续工程项

富文本拆分本身和 Block 重定向已可用，仍有两项长期改进：

1. 抽取 Markdown/Tiptap 共用的事务、附件和撤销服务，减少两套运行时路由重复；
2. 富文本协同升级后，为拆分事务补充格式感知的 CRDT 同步协议。
