# 用户 AI 配置完全隔离设计

## 目标

每个登录用户拥有独立的 AI 配置。任何用户对服务商、API 地址、API Key、模型、配置方案、Embedding 配置或启用开关的读取和修改，都不能影响其他用户。

## 数据模型

新增 `user_ai_settings` 表：

- `userId TEXT NOT NULL`，外键关联 `users(id)`，用户删除时级联删除配置。
- `key TEXT NOT NULL`。
- `value TEXT NOT NULL DEFAULT ''`。
- `updatedAt TEXT NOT NULL`。
- 以 `(userId, key)` 为主键，并为 `userId` 建索引。

表中仅存放 AI 相关配置，系统级配置继续保留在 `system_settings`。API Key 仍按现有规则只在服务端保存，对外返回掩码。

## 隔离范围

以下数据全部按用户隔离：

- 对话服务商、API 地址、API Key、模型。
- AI 配置方案列表和当前启用方案。
- Embedding 地址、API Key、模型。
- 手动 AI 配置启用开关及关闭时的备份值。

工作区不拥有共享 AI 配置。用户在个人空间或工作区发起 AI 请求时，始终使用发起请求者的个人配置。

## 访问边界

增加统一的用户 AI 配置访问层，所有读取、批量读取、写入和删除都必须接收非空 `userId`。业务代码不再直接从 `system_settings` 读取 `ai_*`。

以下调用链必须改为显式传递当前用户：

- `/api/ai` 下的设置、模型发现、连接测试、对话、文档解析、知识库和批处理接口。
- `/api/user-preferences/ai-profiles` 配置方案接口。
- `/api/user-preferences/ai-reliable` 状态、启用开关和问答接口。
- 任务模块中的 AI 调用。
- Embedding 查询与后台索引；后台索引根据笔记所有者读取配置。

缺少认证用户时返回 401，不允许回退到全局配置或其他用户配置。内存缓存如存在，缓存键必须包含 `userId`。

## 手动配置开关

开关按用户独立生效。用户关闭手动配置时，只备份和清空该用户的有效配置；其他用户不受影响。防止关闭后写回配置的保护逻辑也按 `(userId, key)` 判断。

## 升级迁移

新增数据库迁移创建 `user_ai_settings`。升级时：

1. 找出所有现有 `role = 'admin'` 的用户。
2. 将旧 `system_settings` 中允许迁移的 AI 配置复制给每个现有管理员；复制完成后各管理员拥有互不关联的独立副本。
3. 普通用户不复制旧配置，首次进入时使用空配置和默认模型。
4. 删除已迁移的旧全局 AI 配置键，运行时代码不再读取这些键。

允许迁移的键采用明确白名单，包含有效配置、配置方案、Embedding 配置、启用开关和关闭备份；不迁移其他系统设置。

SQLite 迁移、初始 schema 和 PostgreSQL 基础 schema 使用相同的数据模型和约束。

## 前端行为

前端接口形状和设置界面保持不变。认证请求天然绑定当前账号；切换账号后重新读取服务端配置，不复用上一个账号的配置状态。配置列表、掩码状态和启用状态均以当前接口响应为准。

## 验证

- 两个普通用户分别创建、修改、启用和删除配置方案，互相不可见。
- 用户 A 修改对话或 Embedding 配置后，用户 B 的设置和 AI 调用参数不变。
- 用户 A 关闭手动配置不影响用户 B。
- 工作区内两个成员分别发起 AI 请求时，各自使用自己的配置。
- 后台 Embedding 为笔记索引时使用笔记所有者的配置。
- 升级后旧配置只出现在管理员账户，普通用户为空，旧全局 AI 键被移除。
- 所有设置响应继续掩码 API Key，任何用户都无法读取其他用户的密钥。
