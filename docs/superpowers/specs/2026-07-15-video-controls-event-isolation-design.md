# 编辑器视频控件事件隔离设计

## 背景

PC 桌面版富文本编辑器中，上传后插入的视频可以播放，但原生控制条上的暂停、全屏和下载菜单没有响应。附件抽屉中的视频预览不经过 Tiptap NodeView，控件可以正常交互。

## 根因

编辑器视频节点同时启用了 `atom`、`selectable` 和 `draggable`。Tiptap React NodeView 的默认 `stopEvent` 会把可选择节点上的 `mousedown` 交给 ProseMirror。事件目标是原生 `<video>` 时，ProseMirror 的节点选择处理打断了 Chromium/Electron 视频控件所需的完整指针事件序列。

## 目标

- 视频原生控制条可正常暂停、全屏和打开下载菜单。
- 视频节点自定义工具栏中的复制、打开、下载和删除仍可交互。
- 点击视频节点外围仍可选中节点，拖拽、删除等编辑能力保持不变。
- iframe 视频现有的“首次点击选中、再次点击交互”行为保持不变。

## 设计

在 `VideoExtension` 的 `ReactNodeViewRenderer` 选项中配置专用 `stopEvent` 判断函数：

- 事件来自原生 `<video>` 元素或其浏览器控件时，返回 `true`，阻止 ProseMirror 接管事件。
- 事件来自视频自定义工具栏时，返回 `true`，保证链接和按钮不被节点选择逻辑干扰。
- 其他位置返回 `false`，继续使用 ProseMirror 的节点选择与拖拽行为。

自定义工具栏增加稳定的数据属性作为事件边界，不依赖样式类名或显示文案。实现不增加新的控件、不改变视频布局，也不修改全局编辑器事件处理。

## 事件流

1. 用户操作视频控制条或工具栏。
2. NodeView 的 `stopEvent` 根据事件目标判断是否属于交互区域。
3. 属于交互区域时，事件留给浏览器原生视频控件或对应按钮处理。
4. 不属于交互区域时，事件继续交给 ProseMirror，用于节点选择和拖拽。

## 测试

先添加失败回归测试，再实现修复：

- `<video>` 目标的事件必须由 NodeView 截止。
- 视频工具栏及其子元素的事件必须由 NodeView 截止。
- 视频节点普通容器上的事件不能被截止，确保节点仍可选择和拖拽。
- 运行现有 `VideoExtension` 测试、目标 ESLint 和前端生产构建。

## 范围外事项

- 不替换 Chromium 原生视频控制条。
- 不调整视频尺寸、外观或工具栏布局。
- 不改变 iframe 视频的安全沙箱和遮罩策略。
- 不修改附件抽屉中的独立媒体预览组件。
